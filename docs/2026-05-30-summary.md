## 今日完成

### 1. Workflow 新链路与可靠性重构

今天的 workflow 不再只是“能跑一次报告”，而是开始具备真实 CI 审查语义。

关键改进：

- PR `opened`、`synchronize`、`reopened` 自动触发。
- 支持 `workflow_dispatch` 手动触发。
- 使用 base branch reviewer 分析 PR diff，隔离 PR 对审查器自身的破坏。
- 额外 checkout PR head 到 `_pr_head/` 做语法诊断。
- PR head 语法错误时，仍生成 Review 报告和语法诊断；最后一步再让 workflow 失败，避免误合并。
- 上传 Markdown / JSON artifact，JSON 作为后续自动化和 IDE 集成的数据源。
- 高置信、带行号建议发布为 inline review comments。
- summary comment 使用 bot marker 自动更新，避免同一 PR 多次提交后评论刷屏。

这解决了一个关键问题：

```text
报告能否生成 != PR 是否可以合并
```

坏 PR 的理想表现是：

```text
报告生成：是
问题说明：清楚
workflow 最终状态：失败
merge 门禁：阻止
```

### 2. ReviewMeta 与 FixTracking 生命周期基础

今天补齐了 Review 生命周期管理的基础模型：

```text
ReviewMeta
FixTrackingItem
```

ReviewMeta 用于说明报告来源：

- `reviewed_commit`
- `trigger_event`
- `workflow_run_url`
- `updated_at`
- `review_mode`

这些字段会出现在报告顶部，使报告具备可追溯性。

FixTrackingItem 用于未来追踪旧建议：

```text
previous_title
file_path
previous_line
status
detail
```

这为后续“新 commit 推送后，判断上次建议是否已修复”打基础。

今天也验证并修复了 ReviewMeta 传参链路：

- CLI 已支持 `--reviewed-commit`、`--trigger-event`、`--workflow-run-url`。
- Markdown 已支持渲染审查元信息。
- workflow 中曾因 `--help | grep` 探测误判导致参数为空。
- 修正方向为直接传递 ReviewMeta 参数。

### 3. 评论自动更新与同 PR 重跑体验

今天明确了 PR comment 的生命周期目标：

```text
同一个 PR 多次 push / rerun workflow
-> AI Review 重跑
-> 更新同一条 bot summary comment
-> 保留页面整洁
```

已实现的基础：

- summary comment 中包含固定 marker：

  ```text
  <!-- ai-pr-review-bot -->
  ```

- workflow 通过 GitHub API 查找已有 bot comment。
- 找到后 update，找不到才 create。
- 评论中保留 artifact 链接和报告摘要。

这使 GitHub PR 页面从“每次跑都新增一条报告”变为“持续维护一条当前最新报告”。

### 4. Context Pack 轻量项目记忆

今天完成了轻量项目记忆方向，避免模型只看 diff 而不了解项目约定。

新增/更新内容包括：

```text
docs/review-guide.md
docs/functions-index.md
tools/generate_functions_index.py
src/analyzer/context_builder.py
tests/test_context_builder.py
```

Context Pack 的定位：

- 不做重型 RAG。
- 不引入复杂向量数据库。
- 使用轻量、可解释、可维护的项目上下文。

当前上下文来源：

- README 项目概览。
- docs/architecture.md 架构说明。
- docs/review-guide.md 项目 Review 约定。
- docs/functions-index.md 自动生成的函数索引。

作用：

- 帮助模型理解哪些模块高风险。
- 让模型知道本项目的 CI / fallback / report 约定。
- 减少“只看局部 patch 导致误判”的情况。

### 6. VS Code 插件与 VSIX 交付

今天 VS Code 插件从基础 MVP 演进为可安装、可展示、能回到本地代码的 IDE 原型。

当前目录：

```text

```

当前安装包：

```text

```

插件能力：

- 自动识别当前 workspace git branch。
- 使用 `gh` 查找当前分支对应 open PR。
- 拉取 GitHub bot summary comment 和 inline review comments。
- 将 inline suggestions 映射为 VS Code Problems diagnostics。
- 提供 Review Panel。
- Panel 右侧渲染 Markdown summary report。
- Panel 左侧展示 inline suggestions。
- `Open Code` 和文件路径可跳转本地代码。
- CodeLens 在有建议的代码行上方显示 `AI Review: HIGH 92% - ...`。
- 支持手动加载 `pr-review.json`。
- 支持 Clear Diagnostics。

今天还修复了真实安装后 `AI Review: no PR` 的问题：

- 终端能通过 `gh pr list --head <branch>` 找到 PR。
- 旧插件没有指定 workspace root，导致扩展进程可能在错误目录执行 git/gh。
- 修复为使用 VS Code 当前 workspace 根目录执行 fetch。
