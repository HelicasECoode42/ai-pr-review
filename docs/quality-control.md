# 误报与漏报控制

## 误报控制

AI Review 最常见的问题是“看起来合理但没有证据”。本项目通过以下方式降低误报：

- 只允许对 changed lines 生成行级建议
- 每条建议必须给出 `reason` 和 `recommendation`
- 低置信度建议默认不进入 GitHub 评论，只进入报告附录
- 风格类建议默认降权
- 规则命中作为模型证据，不让模型完全自由发散
- 每个文件限制建议数量，避免重复评论

## 漏报控制

完全依赖模型可能漏掉确定性风险，因此加入规则扫描：

- SQL 注入风险
- shell 命令注入风险
- eval/exec 风险
- token/password/secret 日志泄露
- 鉴权代码变更
- 测试删除或跳过
- 异常吞掉

规则扫描不直接等价于最终结论，只作为风险候选和模型输入证据。

## 结果分级

severity：

- `critical`：可能导致安全事故、数据丢失、权限绕过
- `high`：可能导致线上故障或严重业务错误
- `medium`：明显 bug 风险、测试缺口、边界条件问题
- `low`：维护性、可读性、局部改进

confidence：

- `0.85-1.0`：证据强，可作为 Review comment
- `0.65-0.85`：建议人工确认后评论
- `<0.65`：默认只进入报告，不自动评论

## 人机协作边界

这个工具不替代 Reviewer。它的定位是：

- 帮 Reviewer 快速建立 PR 心智模型
- 提醒高风险区域
- 给出可操作的问题候选
- 将重复性检查自动化

最终 merge 判断仍由人负责。
