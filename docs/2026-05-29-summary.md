# 2026-05-29 阶段性总结与下一步计划

## 当前状态

AI PR Review 项目已完成 MVP，具备基本链路：

```
GitHub API → Diff Parser → Risk Rules → Context Builder → LLM Review → Filter → Report
```

但存在三个结构性问题，阻碍从"能跑"到"可靠好用"的跨越。

---

## 三个核心问题

### 问题 1：Workflow 可靠性不足（自举风险）

**现象：** PR 如果修改了 reviewer 工具自身并引入语法/编码/import 错误，AI Review 命令直接启动失败，只剩一段难读的 Actions 日志。没有结构化报告、没有中文解释、没有 PR 总结。

**根因：** 审核执行器和被审核对象没有解耦 —— reviewer 代码和被评审 PR 代码是同一份。

**影响：** 与"极致便捷"目标冲突。用户需要翻 Actions 日志找 traceback，非技术成员根本看不懂。

### 问题 2：Markdown 报告与代码无联动

**现象：**
- 报告中 `位置` 字段写的是纯文本 `` `src/engine.py:104` ``，不是可点击的 GitHub 链接
- 每条建议的"可复制 GitHub 评论"需要用户手动复制粘贴到 PR 页面
- 没有利用 GitHub Review Comments API 在代码旁边发布行级评论

**根因：** 报告生成层缺少 commit SHA 信息，无法构造 blob URL；未对接 GitHub Review Comments API。

**影响：** 报告是孤立文本，评审者需要手动在 IDE/GitHub 里定位代码，效率低。

### 问题 3：报告可信度不可见 & 代码提取鲁棒性不够

**现象：**
- 当 GitHub API 获取失败（如 rate limit）时，可能生成"整体风险 LOW，建议 0 条"的假空报告
- Patch 上下文被裁剪时只标记了 `context_truncated`，但不说清楚哪些文件被跳过了 AI 分析
- `swallowed-exception` 规则误报极高（demo 报告 11 条命中），正则太宽泛
- 多策略 JSON 解析（3 层 fallback）虽然增强了鲁棒性，但解析失败时 `pass` 没有日志，问题不可追踪
- `from None` 丢失了原始异常链，线上调试困难

**根因：**
- 报告缺少"运行状态"和"分析完整性"声明
- 规则正则没有针对 Python/JS/TS 等不同语言做区别化
- 解析失败链路没有日志记录
- 异常转换丢掉了原始 traceback

**影响：** 用户无法判断报告可信度。一份"0 条建议"的报告可能是真的没问题，也可能是 API 挂了。

## 分工方案

### 成员 A：Workflow 可靠性与基础设施

**主题词：** 评审链路不间断

#### Stage 13：可靠评审执行与降级报告

**目标：** 每个 PR 都尽量产出可读报告，失败时有明确状态而非静默失败。

**任务清单：**

1. **PR 分支 reviewer 优先 + main 稳定版兜底**
   - Workflow 先尝试用 PR 分支 reviewer 运行评审
   - 如果 PR 分支 reviewer 启动失败（Python import/syntax error），保存失败日志摘要
   - 自动 `git checkout main`，用 main 分支稳定版 reviewer 重新评审同一个 PR
   - 报告中标注 reviewer 执行版本（PR / main-fallback）和降级原因

2. **基础健康检查独立阶段**
   - 阶段 1：`python -m compileall src tests` 抓语法/编码错误
   - 阶段 2：`uv run pytest` 抓逻辑错误
   - 阶段 3：AI Review（优先 PR 版，兜底 main 版）
   - 阶段 4：上传 artifact + 发布 PR comment + Step Summary

3. **GitHub Step Summary 输出**
   - 失败时在 Actions Summary 展示降级说明和关键错误摘要
   - 用户点进 Actions 不需要下载 artifact 就能看到概要

4. **禁止假空报告**
   - GitHub API 获取失败时生成失败报告，不生成假成功报告
   - 区分：数据获取失败 vs 真正没有发现问题

5. **失败日志摘要写入报告**
   - 如果 PR 版 reviewer 失败，将 `tail -n 80` 的失败日志写入报告
   - 用户不用翻 Actions 日志即知失败原因

**实现顺序建议：**
1. 先加 compileall / pytest / Step Summary
2. 再加失败日志 artifact
3. 最后实现 PR reviewer → main fallback

**验收标准：**
- 构造一个 PR，故意让 `src/analyzer/risk_rules.py` 出现语法错误
- Workflow 不只剩 Actions 日志；自动用 main 版 reviewer 生成报告
- 报告中明确显示 PR 版 reviewer 失败原因
- GitHub API 403 时生成失败报告，而非空成功报告

#### 后续 Stage 14：代码提取鲁棒性增强

**任务清单：**

1. **多策略 JSON 解析增加日志**
   - 三阶段解析失败时 `logger.debug/warning` 记录失败原因
   - 让"为什么解析失败"可追踪

2. **异常链保留**
   - 所有 `raise X(...) from None` 改为 `from exc`
   - 保留原始 traceback 便于调试

3. **规则正则按语言差异化**
   - `swallowed-exception` 针对 Python（`except ...:\s*pass|...`）和 JS/TS（`catch\s*\(...\)\s*\{\s*\}`）分别写正则
   - 降低误报率

4. **文件编码检测**
   - 读取 patch 前检查文件编码，避免 UTF-8 decode error 导致整个评审崩溃
   - 非 UTF-8 文件标记为 `encoding_skipped` 而非报错

### 成员 B：报告可信度与代码联动

**主题词：** 报告可信、可点击、可追溯

#### Stage 15：报告运行状态 & 分析完整性声明

**目标：** 用户从报告顶部就能判断这份报告能不能信。

**任务清单：**

1. **报告顶部增加"运行状态"块**
   ```markdown
   ## 运行状态

   | 字段 | 值 |
   |------|-----|
   | 评审执行版本 | PR 分支 / main fallback |
   | 执行状态 | 成功 / 降级 |
   | 降级原因 | risk_rules.py:75 UTF-8 编码错误（仅降级时出现） |
   | 报告可信度 | 可用于评审 PR diff，但不能代表 PR 版 reviewer 可运行 |
   ```

2. **报告增加"分析完整性"块**
   ```markdown
   ## 分析完整性

   | 项目 | 状态 | 备注 |
   |------|------|------|
   | PR 元信息获取 | ✅ 成功 | |
   | 变更文件获取 | ✅ 成功 | 11 个文件 |
   | AI 上下文文件 | ⚠️ 部分 | 1 个文件跳过（lockfile） |
   | AI 分析 | ✅ 成功 / ❌ 降级 | |
   | 规则扫描 | ✅ 成功 | 12 条命中 |
   | Patch 上下文 | ⚠️ 裁剪 | 超出 24K token 预算 |
   ```

3. **AI Review 兜底标记**
   - 用 main fallback 时标记 `report_confidence = "fallback"`
   - 与正常报告在视觉上区分（加警告样式）

#### Stage 16：Markdown ↔ 代码联动

**目标：** 点报告能跳到代码，代码旁边能看到 AI comment。

**任务清单：**

1. **可点击的文件链接**
   - 在 `GitHubClient` 或 `PullRequest` 中保存 `head_sha`（PR 最新 commit）
   - 报告中的 `src/engine.py:104` 改为可点击链接：
     ```
     https://github.com/owner/repo/blob/<head_sha>/src/engine.py#L104
     ```

2. **GitHub Review Comments 自动发布（高优先级）**
   - 在 GitHub Action 中调用 `pulls.createReviewComment` API
   - 只对高置信度（≥0.85）的建议自动发布行级评论
   - 需要 `commit_id` + `line` + `side=RIGHT`（新增行）
   - 低置信度建议放入 summary comment 折叠区

3. **PR Comment 附带报告摘要**
   - 更新 summary comment 格式，在顶部加运行状态摘要
   - 高信号建议用表格列出（可点击文件链接 + 严重程度）

4. **"可复制 GitHub 评论"块优化**
   - 保留手动复制，作为自动评论的补充
   - 增加一行："💡 可直接复制到 PR Files Changed 页面发布"

## 产品目标升级

从小目标：
> 能跑 AI Review

升级为大目标：
> 任何 PR 打开后，系统都尽量自动产出可读、可信、可追踪的评审结果。
> 即使 AI 不可用、评论权限不足、PR 分支 reviewer 损坏，也会通过规则扫描、artifact、summary 或 main fallback 给出明确状态，而不是静默失败或生成假报告。

## 优先级排序

| 优先级 | Stage | 负责 | 内容 | 理由 |
|--------|-------|------|------|------|
| P0 | 13 | A | PR/main 双版本兜底 | 解决"没报告只剩日志"的最痛问题 |
| P0 | 15 | B | 运行状态 + 分析完整性声明 | 解决"空报告没人知道"的信任问题 |
| P1 | 16 | B | 可点击链接 + Review Comments API | 从文本报告到可交互报告 |
| P1 | 14 | A | 异常链保留 + 规则正则改进 | 提高工具自身代码质量 |
| P2 | 14 | A | 多语言编码检测 | 边缘场景鲁棒性 |

## 备注

- **为什么不用单纯 main 分支 reviewer：** 如果 PR 正在优化 reviewer，报告看不到新版 reviewer 的能力
- **为什么需要 PR 分支优先：** 验证新版 reviewer 能不能跑，如果被改坏则自动降级
- **最终形态：** GitHub App 服务端稳定运行，读取 PR diff 而非执行 PR 里的代码 —— 这是产品级方案，当前 MVP 用 workflow 双版本兜底作为中间态
